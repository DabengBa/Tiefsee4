# Lessons Learned: Copy To Directory（复制到目录）

日期：2026-01-10  
范围：Tiefsee4「复制到目录」功能（快捷键触发 → 解析源文件 → 冲突处理 → 复制落盘 → Toast 反馈）

## 1) 问题描述（What happened?）

### 1.1 快捷键触发后“无错误但也未复制”
- 现象：按下配置的快捷键后，UI 没有明确失败提示，目标目录也没有新增文件。
- 影响：用户无法依赖该功能做“挑选-收集-归档”的高频操作，且因为无反馈导致排查成本变高。

### 1.2 同名文件再次复制时未按预期重命名
- 现象：配置 `onConflict=renameTimestamp` 后，首次复制成功；再次复制同名文件时没有在目标目录生成时间戳新文件（看起来像“策略没生效”）。
- 影响：覆盖/失败/无结果的不确定性增加，破坏“可重复触发”的使用预期。

### 1.3 “先保存再复制”提示频繁出现
- 现象：仅进行“随窗口缩放”的显示行为时，也会被判定为“已变换”，每次复制都弹出“先保存再复制”。
- 影响：交互干扰明显，用户会倾向于关闭功能或忽略提示。

## 2) 根本原因分析（Why did it happen?）

### 2.1 源文件路径并不总是“真实可复制的本地路径”
使用场景中存在：
- 当前展示项没有有效 `filePath`（例如网络图片/虚拟路径/临时渲染结果）。
- `WV_File.Copy` 只能处理真实存在的本地文件路径；当源路径不可用时，流程如果直接返回或异常被吞，就会出现“无反馈且无落盘”。

### 2.2 冲突重命名用字符串拼接路径，导致目标路径错误
重命名策略生成新路径时曾使用 `${dir}${fileName}` 形式拼接：
- 当 `dir` 不保证以 `\\` 结尾时，会生成错误路径（例如 `D:\迅雷下载\pixivabc_...png`）。
- 结果是：逻辑认为“已生成新 destPath”，但实际复制落点不在目标目录，表现为“没有按要求重命名复制”。

### 2.3 变换检测把“随窗口缩放”当作需要保存的变换
- `zoomRatio` 在“跟随窗口缩放”下会变化，但这通常不应被视作“需要保存再复制”的图像变换。
- 造成频繁弹窗，用户体验退化。

## 3) 解决方案与决策（How did we fix it?）

### 3.1 为“非本地源路径”增加可靠回退：导出当前画面到临时 PNG 再复制
- 在普通模式下当 `filePath` 为空或源文件不存在时：
  - 通过 `tiefseeview.getTransformedCanvasBlob("png")` 导出当前画面；
  - 使用 `WV_File.Base64ToTempFile(base64,"png")` 落到临时文件；
  - 再以临时文件作为复制源执行 `WV_File.Copy`。
- 目标：保证“触发后必有结果（成功/失败 Toast）”，避免静默失败。

### 3.2 冲突重命名路径统一用 `Lib.combine` 生成
- `renameTimestamp` 分支使用 `Lib.combine([dir, fileName])` 拼接，避免路径分隔符问题。
- 额外在复制成功后校验 `WV_File.Exists(destPath)`，避免“Copy 返回成功但文件不存在”的黑洞。

### 3.3 变换检测调整：排除“随窗口缩放”导致的缩放变化
- 仅当 `getIsZoomWithWindow() === false` 时才把 zoom 视为需要保存的变换。

## 4) 预防措施（How to prevent it from happening again?）

### 4.1 代码层面的硬约束（建议立即执行）
- 路径拼接禁止字符串相加：统一使用 `Lib.combine`（或 `Path.Combine`）构建文件/目录路径。
- 对跨层 IO（HostObject 文件操作）加“后置条件校验”：
  - `Copy` 后检查目标文件是否存在；
  - 必须保证失败有可见反馈（Toast 或日志）。
- 对“临时导出”相关 API 返回值做严格判定：
  - `Base64ToTempFile` 返回 `"false"` 视为失败，不可当作路径继续执行。

### 4.2 测试与验证（建议在下一迭代补齐）
- 增加最小手动回归用例清单（写入 spec 或 CI 文档）：
  - 普通模式：本地文件 → 复制成功
  - 普通模式：无本地路径/源不存在 → 导出临时 PNG → 复制成功
  - 冲突策略：overwrite/skip/renameTimestamp
  - Bulk View：多选 + 失败统计 + Toast 占位符
- 若后续引入自动化：
  - 对 `resolveWithTimestamp` 做纯函数测试（输入 dir/base/ext → 输出含分隔符的目标路径）。

### 4.3 流程与评审（建议执行）
- 对“新增跨层能力（Web→C# HostObject）”设立评审检查项：
  - 返回值语义（成功/失败）必须统一
  - JS 侧必须处理异常与 UI 反馈
  - 关键路径必须有可复现的最小验证步骤
